# `whygraph scan` — parallel git + GitHub PR crawlers

## Context

`whygraph scan` already walks the first-parent history of the default branch and stores commit metadata in `.whygraph/whygraph.db` (phase 1, shipped). The next step is a **GitHub PR crawler** that runs **in parallel** with the git crawler so a single `whygraph scan` invocation produces a joined picture: one row per commit on `main`, plus one row per PR on the repo, joinable on `pull_requests.merge_commit_sha = commits.sha`.

The user has redirected the order: **PR evidence comes before LLM-per-commit description.** The LLM phase is deferred until the joined commit + PR data has shaped what we actually need to ask the model.

User-facing flow after this lands:

```
$ whygraph scan
Scanning first-parent history of main in /path/to/repo
[git]    [1/49]  abc1234 first commit
[github] page 1/3 (100 PRs)
[git]    [2/49]  def5678 second commit
[github] page 2/3 (100 PRs)
...
Phase complete: 49 commits inserted, 247 PRs inserted.
Done. Database at /path/to/repo/.whygraph/whygraph.db
```

Decisions settled in conversation:

- **Two parallel crawlers in one `whygraph scan` invocation.** Each is its own thread with its own SQLite connection. SQLite in **WAL mode** so both can write concurrently without blocking. Tables are disjoint (`commits` vs `pull_requests`), so no row contention.
- **`gh api ... --paginate --slurp` for the GitHub side.** No raw HTTP, no extra Python deps, no auth code — `gh`'s existing session handles auth. `--slurp` returns the entire paginated result as a single JSON array.
- **Listing endpoints only.** No per-PR `gh pr view <n>` calls. One subprocess per resource type, regardless of repo size.
- **PRs only in this phase. No issues, no PR↔issue linking.** Both deferred until a real consumer needs them.
- **Single-repo schema.** No `provider` / `repo` columns on the new table — the DB is per-repo. If multi-repo aggregation ever becomes a need, that's a future migration.
- **Linking key: `pull_requests.merge_commit_sha = commits.sha`.** Works for squash-merge and merge-commit workflows. Rebase-merge multi-commit PRs only link via the tip; earlier rebased commits stay unlinked. Accepted as a v1 limitation.
- **Threading, not asyncio or multiprocessing.** Both crawlers are subprocess-bound (`git`, `gh`); the GIL doesn't hurt. Threading is simpler and SQLite handles per-thread connections natively.

## Approach

### Phase 1 (shipped)

`src/whygraph/scan/{__init__.py,db.py,git.py,runner.py}` and the `scan` Click subcommand are already in place. 49 first-parent commits scanned cleanly on this repo, idempotent re-runs verified. Tests in `tests/test_scan_db.py` and `tests/test_scan_git.py`. Not modifying any of this except the runner (to gain a parallel crawler) and the DB module (to add a migration).

### Phase 2 — parallel GitHub PR crawler (this plan)

Add a second crawler that lists PRs from the GitHub repo associated with `origin` and stores them alongside commits. Both crawlers run in parallel within a single `whygraph scan` invocation.

#### Schema migration v2

Add to `src/whygraph/scan/db.py`'s `_MIGRATIONS`:

```sql
CREATE TABLE pull_requests (
  number INTEGER PRIMARY KEY,
  title TEXT NOT NULL,
  body TEXT,
  state TEXT NOT NULL,                       -- 'open' | 'closed'
  draft INTEGER NOT NULL DEFAULT 0,          -- 0/1 boolean
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL,
  closed_at TEXT,
  merged_at TEXT,
  merge_commit_sha TEXT,                     -- joins to commits.sha
  head_sha TEXT NOT NULL,
  head_ref TEXT,
  base_ref TEXT NOT NULL,
  author TEXT,
  html_url TEXT NOT NULL,
  labels TEXT NOT NULL,                      -- JSON array of label names
  fetched_at TEXT NOT NULL
);
CREATE INDEX idx_pull_requests_merge_commit_sha ON pull_requests(merge_commit_sha);
CREATE INDEX idx_pull_requests_state ON pull_requests(state);
```

The `Database._migrate()` machinery already supports versioned migrations — bump `SCHEMA_VERSION` to 2, add v2 statements to `_MIGRATIONS`. Existing `commits` and `scan_state` tables are untouched.

Also add a one-line `PRAGMA journal_mode=WAL` in `Database.__init__` (right after the `PRAGMA foreign_keys = ON`). WAL has to be set per-database, not per-connection, so once one connection sets it, the file stays in WAL mode for all future connections.

#### New module: `src/whygraph/scan/github.py`

Pure-stdlib subprocess wrapper around `gh`. Public surface:

```python
@dataclass(frozen=True)
class PullRequest:
    number: int
    title: str
    body: str | None
    state: str
    draft: bool
    created_at: str
    updated_at: str
    closed_at: str | None
    merged_at: str | None
    merge_commit_sha: str | None
    head_sha: str
    head_ref: str | None
    base_ref: str
    author: str | None
    html_url: str
    labels: list[str]


class GitHubError(RuntimeError):
    pass


def detect_repo(repo_root: Path) -> tuple[str, str] | None:
    """Return (owner, name) if origin is a GitHub remote, else None."""
    # Parse `git config --get remote.origin.url`. Match
    # https://github.com/{o}/{n}(.git)?
    # git@github.com:{o}/{n}(.git)?
    # ssh://git@github.com/{o}/{n}(.git)?


def list_pull_requests(owner: str, name: str) -> list[PullRequest]:
    """Run `gh api '/repos/{o}/{n}/pulls?state=all&per_page=100' --paginate --slurp`
    and parse each item into a PullRequest. Returns all PRs, sorted by number ascending."""
```

Implementation notes:

- `gh api ... --paginate --slurp` returns a single JSON array. With `--slurp`, even paged responses come back as one parseable blob.
- Subprocess timeout: none for the gh call itself (a 5000-PR repo can take a couple of minutes); but use `subprocess.run` with `check=False` and surface non-zero returncodes as `GitHubError` with the stderr included.
- `gh` not installed → distinguish from "gh installed but not authenticated" → distinguish from network/API error. Three separate error messages.
- Auth check at the start: `gh auth status` (single subprocess) before kicking off the slurp. If unauthenticated, exit early with "run `gh auth login`".

#### New `scan/db.py` accessor

```python
def upsert_pull_request(self, pr: PullRequest) -> bool:
    """Insert if new, update fetched_at if existing. Returns True if newly inserted."""
```

Use `INSERT ... ON CONFLICT(number) DO UPDATE SET ...` — PRs change over time (state, body edits, labels), so an upsert that refreshes mutable fields is more useful than a pure insert-if-absent.

#### Runner update: `src/whygraph/scan/runner.py`

Restructure into two crawler functions plus a parallel dispatcher:

```python
def _git_crawler(repo_root: Path, branch: str, db_path: Path) -> dict:
    """Phase 1 logic, factored out. Owns its own Database connection."""
    with db_module.Database(db_path) as db:
        ...
    return {"inserted": ..., "skipped": ..., "total": ...}


def _github_crawler(repo_root: Path, db_path: Path) -> dict:
    owner_name = github.detect_repo(repo_root)
    if owner_name is None:
        return {"skipped_reason": "origin is not a GitHub remote"}
    owner, name = owner_name
    prs = github.list_pull_requests(owner, name)
    with db_module.Database(db_path) as db:
        inserted = sum(1 for pr in prs if db.upsert_pull_request(pr))
    return {"inserted": inserted, "total": len(prs)}


def run_scan(*, skip_git: bool = False, skip_github: bool = False, ...) -> int:
    root = git_module.repo_root(...)
    branch = git_module.default_branch(root)
    db_path = db_module.default_db_path(root)
    # Materialize schema once on the main thread before crawlers open their own connections.
    with db_module.Database(db_path) as db:
        pass

    futures: dict[str, Future] = {}
    with ThreadPoolExecutor(max_workers=2, thread_name_prefix="whygraph-crawler") as ex:
        if not skip_git:
            futures["git"] = ex.submit(_git_crawler, root, branch, db_path)
        if not skip_github:
            futures["github"] = ex.submit(_github_crawler, root, db_path)
        # Wait, collect results, surface any exceptions.
    ...
    return 0
```

The "materialize schema once on the main thread" step matters: if both threads try to apply migrations simultaneously, you get a race. Doing it once up front (and switching to WAL) means crawler threads only do reads/writes, never DDL.

Progress output: each crawler echoes prefixed lines (`[git]`, `[github]`) so the user can see both moving. Click's `echo` is thread-safe (it writes to stdout via Python's IO layer, which holds the GIL during write).

## Verification

End-to-end (manual):

1. **Both crawlers, real repo:**
   ```bash
   cd ~/Projects/demos/whygraph
   rm -rf .whygraph
   uv run whygraph scan
   ```
   Expect interleaved `[git]` and `[github]` progress. Then:
   ```bash
   sqlite3 .whygraph/whygraph.db "SELECT COUNT(*) FROM commits"      # 49
   sqlite3 .whygraph/whygraph.db "SELECT COUNT(*) FROM pull_requests" # however many PRs
   sqlite3 .whygraph/whygraph.db "
     SELECT c.sha, c.subject, p.number, p.title
     FROM commits c JOIN pull_requests p ON p.merge_commit_sha = c.sha
     LIMIT 5
   "
   # expect commits joined to their PRs
   ```

2. **Idempotent re-run:** `uv run whygraph scan` again. Same row counts. PRs that have changed state (e.g., a PR closed since last scan) should have updated `state` / `closed_at` / `fetched_at` due to upsert-on-conflict.

3. **Skip flags:**
   - `uv run whygraph scan --no-github` → only git crawler runs, only `commits` updated.
   - `uv run whygraph scan --no-git` → only github crawler runs, only `pull_requests` updated.
   - Both flags together: prints "nothing to do" and exits 0.

4. **Non-GitHub origin:** in a repo whose `origin` points at GitLab/Bitbucket/none, the github crawler short-circuits with a clear "origin is not a GitHub remote" message. The git crawler still runs.

5. **Unauthenticated `gh`:** with `gh auth logout` first, `whygraph scan` reports "run `gh auth login`" and skips the GitHub crawler (or aborts — TBD which is friendlier; default to "skip with warning, exit 0" so a CI-without-gh environment still gets git data).

## What this plan deliberately does NOT do

- **No issues table.** Deferred until a consumer needs it.
- **No PR ↔ Issue linking.** Same.
- **No per-PR `gh pr view <n>` calls.** Listing only, by user constraint.
- **No `provider` / `repo` columns.** Schema is per-repo until a real multi-repo case forces a migration.
- **No retry logic on `gh` failures.** First version surfaces the error; if rate-limit retries become necessary, add then.
- **No LLM yet.** That moves to phase 3 after this lands.
- **No GraphQL.** REST listing is enough; GraphQL would be needed only for the linked-issues sidebar info, which we're skipping.
- **No commit-to-node linking.** Still phase 4+; the new schema doesn't try to anticipate it.
